# OpsClaw MVP 产品需求文档（PRD v1.0）

## 1. 文档信息

**产品名称：** OpsClaw  
**版本：** MVP v1.0  
**产品形态：** 基于 xterm.js 的 AI 运维终端桌面应用 / Web 应用原型  
**文档目标：** 明确 OpsClaw 第一阶段 MVP 的产品定位、目标用户、核心场景、功能范围、非功能要求与实施边界，指导后续设计与开发。

---

## 2. 产品背景

当前大型信息运维中心在日常工作中，通常会同时面对以下问题：

1. **节点数量多，环境复杂：** 机器分散在不同业务、不同网络区、不同集群和不同责任域，日常连接与管理成本高。
2. **工具割裂：** 终端连接、批量执行、巡检脚本、异常分析、结果汇总、告警通知往往分散在多个工具中完成，切换频繁。
3. **巡检效率低：** 大量巡检仍依赖人工登录节点执行固定命令，效率低且容易遗漏。
4. **AI 能力未真正嵌入运维流程：** 目前不少 AI 工具停留在"聊天问答"层面，无法成为运维人员高频使用的工作台。
5. **自动化与安全之间矛盾突出：** 越是希望 AI 介入执行，越需要更严格的权限控制、命令边界、审计与审批机制。

MiniClaw 已经验证了"AI + Skill + 节点连接 + 巡检分析"的可行性，但随着场景深入，通用 Agent 形态难以承载高频、强约束、可审计、可批量、可调度的运维场景。

因此，需要设计一个面向运维中心的独立产品：**OpsClaw**。

OpsClaw 的核心定位不是一个聊天机器人，而是一个：

> **像 Xshell 一样可以直接使用，同时具备 AI 辅助能力、批量执行能力、巡检脚本能力和异常分析能力的运维终端工作台。**

---

## 3. 产品定位

### 3.1 产品定义

OpsClaw 是一个面向运维中心场景的 AI 原生运维终端，基于 xterm.js 构建终端交互壳层，提供：

- 节点管理
- 分组管理
- 单机终端
- 批量命令执行
- 巡检脚本配置与执行
- 巡检结果快照保存
- AI 输出解释与风险总结
- 告警通知
- 基础审计能力

### 3.2 核心价值

OpsClaw 在 MVP 阶段重点解决三个问题：

1. **让运维人员有一个真正可用的统一入口**  
   不是聊天工具，而是日常就能打开使用的运维终端。

2. **把高频重复巡检动作结构化、模板化、批量化**  
   让人工登录逐台检查，升级为脚本化巡检与结果聚合。

3. **让 AI 成为解释器、建议器和风险放大镜，而不是失控的执行者**  
   AI 在 MVP 中主要负责解释、总结、建议和告警辅助，不直接拥有高危自动执行权限。

## 4. 目标用户

### 4.1 核心用户

1. **一线运维工程师**  
   日常登录机器、检查服务状态、执行命令、查看日志、进行故障初排。

2. **值班运维 / NOC 人员**  
   需要定时巡检节点状态，发现异常后快速汇总并通知。

3. **运维负责人 / 平台管理员**  
   需要管理节点、脚本、分组、权限、审计记录。

### 4.2 典型使用特征

- 管理节点数量多
- 需要批量执行类似命令
- 经常有重复的检查动作
- 需要结构化保存历史结果
- 需要快速得出"哪些机器异常、异常在哪里"
- 对 AI 有兴趣，但更关注安全、稳定、审计和可控

## 5. MVP 目标与边界

### 5.1 MVP 目标

在第一阶段交付一个可试用、可演示、可投入小范围试点的产品，满足以下目标：

1. 支持基础的节点连接和终端使用
2. 支持节点分组与批量命令执行
3. 支持巡检脚本配置和定时执行
4. 支持保存巡检快照与查看历史结果
5. 支持 AI 对结果进行解释、总结和风险提示
6. 支持邮件 / 飞书等基础告警
7. 支持基础审计与命令执行留痕

### 5.2 MVP 不做的内容

为控制范围，以下内容明确不纳入 MVP：

- RDP / VNC / Telnet 等多协议支持
- 完整 SFTP 文件管理器
- 自定义插件市场
- AI 全自动修复闭环
- 复杂审批流系统
- 多租户 SaaS 化能力
- 高级 CMDB 深度集成
- 自动资产发现
- 全面 SOP 编排平台

## 6. 典型使用场景

### 场景 1：单节点日常登录与命令检查

用户打开 OpsClaw，在节点列表中选择目标机器，进入终端页面，执行 `top`、`df -h`、`systemctl status` 等命令，并查看输出。AI 可对输出进行解释。

### 场景 2：批量执行检查命令

用户选择某个分组，例如"生产 Redis 集群"，批量执行 `df -h` 或自定义脚本，系统返回每台机器的执行结果，并在汇总视图中突出异常节点。

### 场景 3：定时巡检

用户为某个分组配置巡检脚本和执行时间，例如工作日 09:00~18:00 每 30 分钟执行一次。系统自动运行脚本、保存快照、对异常进行分析并发送告警。

### 场景 4：异常解释与告警通知

巡检结果中发现某节点磁盘使用率高、某服务失败、某日志存在关键错误。系统调用大模型生成摘要，标注风险级别，并发送邮件或飞书通知。

### 场景 5：自然语言辅助理解

用户选中某批机器执行完命令后，可以点击"AI 总结"，由系统总结：

- 共有多少台机器异常
- 异常类型有哪些
- 哪些节点最需要优先处理
- 后续建议检查哪些命令

## 7. 功能需求

### 7.1 节点管理

#### 7.1.1 功能说明

OpsClaw 需要支持基础的节点管理，作为所有执行与巡检的前提。

#### 7.1.2 子功能

1. **新增节点**
   - 支持录入节点名称、主机地址、端口、用户名、认证方式、备注、标签、所属分组

2. **编辑节点**

3. **删除节点**

4. **测试连接**

5. **导入节点**
   - 支持 CSV / JSON 导入

6. **导出节点**
   - 支持导出基础节点信息（不导出敏感凭证明文）

7. **节点标签管理**
   - 如环境、业务、角色、机房等

#### 7.1.3 数据字段建议

- `node_id`
- `node_name`
- `host`
- `port`
- `username`
- `auth_type`（password / private_key）
- `credential_ref`
- `tags`
- `group_id`
- `description`
- `created_at`
- `updated_at`

#### 7.1.4 需求说明

- 节点必须支持分组与标签，便于后续筛选与批量执行
- 凭证信息不以明文展示
- MVP 阶段支持手动录入和文件导入

### 7.2 分组管理

#### 7.2.1 功能说明

用于组织大量节点，便于批量执行与巡检。

#### 7.2.2 子功能

1. 创建分组
2. 编辑分组
3. 删除分组
4. 向分组中添加 / 移除节点
5. 查看分组下节点数量与状态

#### 7.2.3 分组模型

**MVP 阶段支持：**
- 静态分组

**后续版本可扩展：**
- 动态标签分组

### 7.3 终端能力（基于 xterm.js）

#### 7.3.1 功能说明

提供单节点交互式终端能力，是 OpsClaw 的核心入口。

#### 7.3.2 子功能

1. 打开节点终端会话
2. 输入命令并实时查看输出
3. 支持常见 ANSI 转义显示
4. 支持复制粘贴
5. 支持终端会话断开提示
6. 支持基础终端设置
   - 字体大小
   - 主题（浅色 / 深色）
   - 滚动缓冲区长度
7. 支持命令历史记录

#### 7.3.3 设计要求

- 基于 xterm.js 构建前端终端显示层
- 服务端负责 SSH 连接与流式输出转发
- 终端体验优先保证"够用、稳定、清晰"，不追求首版做到专业终端软件全部细节

### 7.4 批量命令执行

#### 7.4.1 功能说明

用户可对某个分组或多台节点执行同一命令，系统统一收集并汇总结果。

#### 7.4.2 子功能

1. 选择节点 / 分组
2. 输入命令或选择脚本模板
3. 设置并发数
4. 设置超时时间
5. 执行并展示进度
6. 查看每台机器执行结果
7. 展示成功 / 失败统计
8. 支持结果导出

#### 7.4.3 结果展示要求

- 结果按节点聚合展示
- 支持高亮失败节点
- 支持折叠相同结果
- 支持过滤仅查看异常结果

#### 7.4.4 安全要求

- 批量执行前需明确显示目标节点数
- 对高危命令进行拦截或二次确认

### 7.5 巡检脚本管理

#### 7.5.1 功能说明

支持用户配置标准巡检脚本，用于定时或手动执行。

#### 7.5.2 子功能

1. 创建巡检脚本
2. 编辑巡检脚本
3. 删除巡检脚本
4. 绑定目标分组
5. 脚本版本记录（MVP 可简化为更新时间与版本号）
6. 手动执行脚本

#### 7.5.3 脚本内容约束

**MVP 阶段建议优先支持：**
- shell 脚本
- 受限命令模板

**不建议首版支持过于自由的多语言复杂执行。**

#### 7.5.4 推荐脚本类型

- 系统资源巡检
- 服务状态巡检
- 日志关键字巡检
- 磁盘与 inode 巡检
- 时间同步 / 证书检查

### 7.6 巡检调度

#### 7.6.1 功能说明

支持为巡检脚本配置定时执行规则。

#### 7.6.2 子功能

1. 创建巡检任务
2. 选择巡检脚本
3. 选择目标分组 / 节点
4. 设置 cron 表达式或简化调度规则
5. 设置工作时间窗口
6. 设置是否启用
7. 查看最近执行记录

#### 7.6.3 MVP 要求

- 支持基础 cron 调度
- 支持工作日、时间段限制
- 支持失败重试次数配置
- 支持暂停 / 启用任务

### 7.7 巡检快照与历史记录

#### 7.7.1 功能说明

每次巡检执行后保存结果，便于回溯与比较。

#### 7.7.2 快照内容

每次快照至少保存：

- 执行时间
- 执行对象
- 脚本名称与版本
- 原始输出
- 执行状态
- AI 分析结果（如有）
- 风险等级

#### 7.7.3 子功能

1. 查看巡检历史列表
2. 查看单次巡检详情
3. 按节点 / 分组 / 脚本 / 时间筛选
4. 导出巡检结果

#### 7.7.4 MVP 边界

首版可不做复杂可视化趋势图，重点保证可查、可追溯、可导出。

### 7.8 AI 分析能力

#### 7.8.1 功能定位

MVP 阶段的 AI 主要承担以下角色：

- 输出解释器
- 异常总结器
- 风险提示器
- 下一步建议提供者

**而不是完全自动执行者。**

#### 7.8.2 子功能

1. 单次命令输出解释
2. 批量结果总结
3. 巡检结果风险摘要
4. 异常节点优先级排序建议
5. 后续排查命令建议

#### 7.8.3 输入来源

- 单节点命令输出
- 批量执行汇总结果
- 巡检脚本原始输出
- 系统规则引擎初筛结果

#### 7.8.4 输出示例

- 共有 3 台节点存在磁盘使用率异常，其中 1 台达到高风险
- 两台机器 systemd 存在 failed services，建议优先检查 xxx 服务
- 某节点 dmesg 中出现 I/O error，建议进一步排查磁盘健康状态

#### 7.8.5 成本控制原则

- 优先由规则引擎判断是否需要调用 AI
- 正常结果不强制调用模型
- 异常结果或用户主动点击"AI 总结"时再调用模型

### 7.9 告警通知

#### 7.9.1 功能说明

当巡检任务发现异常时，可通过配置渠道进行通知。

#### 7.9.2 子功能

1. 配置通知渠道
2. 配置告警触发条件
3. 发送告警消息
4. 记录发送历史

#### 7.9.3 MVP 支持渠道

- 邮件
- 飞书 Webhook

#### 7.9.4 告警内容建议

- 时间
- 巡检任务名称
- 目标分组 / 节点
- 风险等级
- 异常摘要
- 跳转查看详情链接

### 7.10 审计与安全控制

#### 7.10.1 功能说明

记录关键操作，保证执行可追溯。

#### 7.10.2 审计范围

MVP 至少记录：

- 节点新增 / 编辑 / 删除
- 批量命令执行
- 巡检任务创建 / 修改 / 删除
- 巡检任务执行记录
- AI 分析触发记录
- 告警发送记录

#### 7.10.3 安全控制要求

1. 凭证加密存储
2. 高危命令拦截或二次确认
3. AI 不直接执行高危动作
4. 命令执行记录可追溯到用户

#### 7.10.4 高危命令示例

- `rm -rf /`
- `reboot`
- `shutdown`
- `mkfs`
- 批量 `systemctl stop`

MVP 阶段可通过简单规则匹配实现拦截。

## 8. 非功能需求

### 8.1 安全性

- 凭证不得明文落库
- 必须支持最基本的权限隔离
- 关键执行需记录操作者
- AI 相关执行必须受策略限制

### 8.2 性能

- 单节点终端操作应接近实时
- 批量执行需支持可配置并发
- 巡检任务执行应可控、可限流

### 8.3 稳定性

- SSH 会话异常断开需提示用户
- 定时任务失败需有记录
- AI 服务不可用时不影响核心执行链路

### 8.4 可扩展性

- 后续能扩展更多协议和更多通知渠道
- 后续能扩展动态分组与审批流
- 后续能扩展插件机制

## 12. 风险与约束

### 12.1 产品风险

1. 用户容易把它期待成"全自动运维平台"，但 MVP 实际定位是"AI 增强运维终端"
2. 若功能过多，容易拖慢交付周期
3. 若过早开放高危自动执行，安全风险高

### 12.2 技术风险

1. SSH 会话与终端交互细节可能有兼容问题
2. 批量执行在大规模节点下可能出现并发压力
3. AI 分析成本需控制
4. 巡检输出若缺乏结构化，后续分析效果受限

### 12.3 应对策略

- MVP 聚焦高频核心能力
- 优先稳定执行链路，再增强 AI
- 通过规则引擎先做异常初筛
- 设计可扩展数据结构，为后续升级留接口

## 13. 验收标准（MVP）

满足以下条件可视为 MVP 达到可试点状态：

1. 用户可完成节点新增、编辑、导入、分组管理
2. 用户可打开节点终端并执行基础命令
3. 用户可对一个分组执行批量命令并查看结果
4. 用户可配置巡检脚本并手动执行
5. 用户可创建定时巡检任务并成功生成快照
6. 用户可查看历史巡检记录
7. 用户可对异常结果触发 AI 总结
8. 用户可在异常时收到邮件或飞书告警
9. 系统可记录基础审计日志

## 15. 一句话总结

OpsClaw MVP 不是要一次性做成一个全自动智能运维平台，而是先做成一个：

> **真正能被运维人员每天打开使用的、基于 xterm.js 的 AI 增强型运维终端工作台。**

它首先是一个可用的终端，其次是一个可批量执行、可巡检、可留痕、可告警的运维工具，最后才是一个逐步长出智能能力的 Ops AI 产品。
