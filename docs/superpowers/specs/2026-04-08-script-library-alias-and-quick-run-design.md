# OpsClaw 脚本库别名与终端快捷执行设计

> 日期：2026-04-08
> 状态：已确认设计，待进入 implementation planning

## 1. 背景

OpsClaw 当前已经有脚本库首版能力，支持：

- 全局脚本与节点脚本
- 节点脚本覆盖全局脚本
- plain 脚本直接发送到当前会话
- template 脚本填写变量后发送到当前会话

但当前版本有两个明显问题：

1. 脚本库前端列表使用大卡片展示，单条脚本占用面积过大。当脚本数量达到 20 到 30 条时，可浏览性和切换效率明显下降。
2. 脚本库虽然能“沉淀脚本”，但没有形成“终端内的高频快捷入口”。用户仍需要先打开脚本库、再搜索、再点执行，无法形成接近日常命令输入的工作流。

本次迭代目标不是新增一套新的脚本系统，而是把现有脚本库产品化，使其真正变成：

- 能高密度浏览的脚本资产列表
- 能在终端中通过 `x alias` 快捷执行的脚本入口

## 2. 目标

本次设计要达成：

- 为脚本新增独立的 `alias` 字段，作为终端快捷执行入口
- 在终端中支持 `x + 空格 + alias` 的快捷脚本模式
- 支持延迟 300ms 的本地补全推荐，避免影响用户正常输入字母 `x`
- 用户直接输入完整 `x alias` 并回车时，若精确匹配，则直接执行
- `plain` 脚本直接执行到当前 session
- `template` 脚本通过前端原生参数卡片收集变量后再执行
- 脚本库列表从“大卡片模式”改为“高密度列表模式”
- alias 解析遵循“当前节点优先覆盖全局”的既有脚本作用域规则

## 3. 非目标

本次不包含：

- 将快捷脚本同步为远端 shell alias / function
- 多脚本批量执行
- 模糊命中后自动执行多个候选
- 为快捷脚本引入审批流或 agent runtime interaction 协议
- 脚本版本管理
- 脚本权限系统

本次只做“本地快捷识别 + 本地补全 + 复用现有脚本执行链路”。

## 4. 设计原则

### 4.1 `key` 与 `alias` 分离

现有 `key` 继续承担脚本稳定业务键职责，用于：

- 节点脚本覆盖全局脚本
- 数据存储与查找
- 兼容已有脚本模型

新增 `alias` 只承担：

- 终端快捷执行入口
- 人类高频记忆和输入

不能让 `key` 同时承担覆盖键和快捷命令名，否则后续产品语义会继续混乱。

### 4.2 快捷模式是本地终端增强，不是远端 shell 能力

`x alias` 的识别、补全、解析和模板变量卡片都发生在 OpsClaw 本地客户端中，不写入远端机器环境，不依赖远端 shell 配置。

### 4.3 终端优先，不打断用户

快捷模式必须服务于“在终端里快速执行脚本”的目标，因此：

- 只在整行开头匹配 `x `
- 推荐浮层只做辅助，不锁终端
- 若未命中 alias，不破坏正常终端输入

### 4.4 继续复用现有执行链路

脚本最终执行仍然走当前的：

- `TerminalWorkspace.sendCommandToActive`
- `SshTerminalPane.sendCommand`
- WebSocket input 注入

不单独实现第二套脚本执行器。

## 5. 用户体验设计

## 5.1 脚本库列表改造

现有脚本库改为高密度双区布局：

1. 左侧或上侧为脚本列表
2. 右侧或下侧为脚本详情与执行区

列表区域不再使用大卡片，而使用高密度行式列表。每行仅展示最关键字段：

- `alias`
- 标题
- 作用域来源
- 类型
- 标签摘要

可选展示：

- 简短描述
- 是否为节点覆盖脚本

设计要求：

- 单屏应能容纳明显多于当前版本的脚本数量
- 用户能快速扫 alias 与标题
- 选中行后详情区展示完整描述、脚本内容、变量输入和执行按钮

### 5.2 新建 / 编辑脚本

新建脚本时，`alias` 改为必填字段。

表单中字段职责如下：

- `key`：稳定业务键，用于覆盖关系
- `alias`：终端快捷入口，用于 `x alias`
- `title`：用户可读标题
- `description`：补充说明
- `kind`：plain / template
- `content`：脚本内容

前端需在编辑时实时提示：

- alias 格式是否合法
- alias 是否与当前作用域内的脚本冲突

### 5.3 终端快捷模式

用户在终端当前输入行满足以下条件时进入快捷脚本模式：

- 整行开头匹配 `x `

示例：

- `x nginx-restart`
- `x deploy_api`

以下情况不触发：

- `echo x foo`
- `sudo x foo`
- 行内其他位置出现 `x `

### 5.4 推荐浮层

当用户输入 `x ` 后，若持续停顿 300ms，出现本地推荐浮层。

浮层行为：

- 根据当前 session 对应节点上下文解析可见脚本
- 优先展示当前节点脚本
- 若当前节点没有匹配，再展示全局脚本
- 支持继续输入 alias、标题、标签过滤
- 回车执行当前高亮项
- `Esc` 关闭浮层
- 正常继续输入时实时更新候选

浮层仅是辅助，不锁输入，不阻止用户直接手输完整 alias。

### 5.5 直接回车执行

若用户完整输入 `x alias` 并回车：

- 若精确匹配到 `plain` 脚本，则直接执行
- 若精确匹配到 `template` 脚本，则弹出参数卡片，填写后执行
- 若未命中任何 alias，则不执行脚本，并给出轻量提示

本次设计中，“未命中时是否将原始 `x alias` 继续发送给远端 shell”不采用兜底透传。默认视为一次本地快捷脚本尝试失败，并在本地提示“未找到脚本别名”，避免向远端注入不存在的 `x` 命令造成混乱。

### 5.6 Help messaging

Help 界面现在明确写出脚本库支持 alias，并提示用户可以在终端通过 `x <alias>` 快速执行脚本，帮助他们在没有阅读文档或设置的情况下也能知道快捷入口的存在。

## 6. 数据模型

## 6.1 ScriptLibraryItem 扩展

在现有脚本模型上新增：

```ts
type ScriptLibraryItem = {
  id: string;
  key: string;
  alias: string;
  scope: 'global' | 'node';
  nodeId: string | null;
  title: string;
  description: string;
  kind: 'plain' | 'template';
  content: string;
  variables: ScriptVariableDefinition[];
  tags: string[];
  resolvedFrom: 'global' | 'node';
  overridesGlobal: boolean;
  createdAt: string;
  updatedAt: string;
};
```

`ScriptLibraryUpsertInput` 同步新增：

```ts
type ScriptLibraryUpsertInput = {
  key: string;
  alias: string;
  scope: 'global' | 'node';
  nodeId: string | null;
  title: string;
  description: string;
  kind: 'plain' | 'template';
  content: string;
  variables: ScriptVariableDefinition[];
  tags: string[];
};
```

## 6.2 Alias 格式

首版 alias 采用保守约束：

- 仅允许小写字母
- 允许数字
- 允许 `-`
- 允许 `_`

建议正则：

```ts
/^[a-z0-9][a-z0-9_-]*$/
```

不允许：

- 空格
- 大写字母
- 点号
- shell 特殊字符

这样可以降低终端快捷歧义。

## 6.3 唯一性规则

alias 的唯一性采用“分层唯一，当前节点优先覆盖全局”：

- 全局脚本层：`alias` 全局唯一
- 节点脚本层：同一 `nodeId` 下 `alias` 唯一
- 解析 `x alias` 时：
  1. 先查当前节点下的 alias
  2. 若不存在，再查全局 alias

这意味着：

- 全局与节点之间允许同 alias 共存
- 当前节点脚本可覆盖同 alias 的全局脚本

`key` 的唯一性和覆盖语义保持不变，不因 alias 引入而修改。

## 7. 解析与执行流程

## 7.1 运行时解析视图

服务端继续提供“已解析后的当前可见脚本列表”，但本次需要新增 alias 字段，并让返回项能够支持终端快速解析。

终端快捷模式只消费“当前 session 对应节点的 resolved scripts”视图，不自己重复实现跨作用域合并逻辑。

## 7.2 快捷命令状态机

终端输入层新增本地状态机：

1. 监听当前输入行变化
2. 若整行以 `x ` 开头，进入快捷脚本模式
3. 启动 300ms 延迟计时器
4. 到时后若仍在快捷模式，则请求或读取当前脚本候选
5. 渲染建议浮层
6. 用户继续输入时更新过滤结果
7. 用户回车时：
   - 若浮层选中项存在，执行该脚本
   - 若无浮层选中项但存在精确 alias 命中，执行该脚本
   - 若无命中，展示未找到提示

## 7.3 执行分流

命中脚本后：

- `plain`
  - 直接取 `content`
  - 调用现有 `sendCommand`
  - 关闭建议浮层

- `template`
  - 不直接执行
  - 打开本地原生参数卡片
  - 校验变量值
  - 使用现有模板渲染逻辑生成最终命令
  - 调用现有 `sendCommand`

## 7.4 本地参数卡片

template 脚本的变量收集卡片不使用 agent runtime 的 `interaction.request` 协议。

原因：

- 这是用户主动触发的本地脚本执行
- 不属于 agent run 的控制流暂停
- 不应污染 agent interaction 状态机

因此本次设计采用脚本库本地 UI 卡片或对话框处理模板变量。

## 8. 前端架构

## 8.1 脚本库列表模型

建议新增或扩展脚本库纯函数模型，用于：

- alias 校验
- alias 冲突提示
- 高密度列表排序
- 搜索同时覆盖 `alias / title / key / description / tags`

### 8.2 终端快捷脚本模型

建议新增独立纯函数模块，例如：

- `terminalQuickScriptModel.ts`

职责：

- 判断当前输入是否进入快捷模式
- 提取 alias 查询串
- 过滤脚本候选
- 计算候选排序
- 识别精确 alias 命中
- 决定是否显示建议浮层

建议不要把这部分逻辑塞进现有 script library model 中，以免把“脚本存储逻辑”和“终端输入交互逻辑”耦合在一起。

## 8.3 终端集成位置

推荐接入点在现有终端运行时输入层，而不是 `sendCommandToActive` 这种最终发送函数上。

原因：

- 补全和推荐需要感知“用户正在输入什么”
- 300ms 延迟建议需要输入态
- `sendCommandToActive` 只负责“最终发送”，看不到输入过程

因此应在终端组件输入事件附近扩展：

- 输入缓冲识别
- 建议浮层
- 回车拦截

## 8.4 现有建议浮层复用

OpsClaw 当前终端已有 suggestion overlay 能力。本次优先复用已有浮层体系，仅扩展其数据源与触发语义：

- 普通建议：保留现状
- `x ` 快捷模式建议：新增一类脚本候选源

不建议为脚本再造一个全新浮层组件。

## 9. 后端架构

## 9.1 数据库存储

脚本表新增 `alias` 列：

```sql
alias TEXT NOT NULL
```

并新增索引与唯一性约束：

- 全局层：`(scope, alias)` 或等价约束
- 节点层：`(scope, node_id, alias)` 或等价约束

由于 SQLite 条件唯一索引更适合表达首版规则，建议使用：

- `global` alias 唯一索引
- `node` + `node_id` + alias 唯一索引

## 9.2 Store 能力

`scriptLibraryStore` 需要新增 / 调整：

- 创建时校验 alias 合法性
- 更新时校验 alias 合法性
- 检查分层唯一性
- 在 `listResolvedScripts(nodeId?)` 中返回 alias

若后续需要按 alias 快速解析，可新增只读帮助函数，但首版前端可直接基于 resolved list 本地匹配，不强制新增单独 API。

## 10. 错误处理

需要覆盖：

1. alias 非法
   - 创建/编辑时内联报错

2. alias 冲突
   - 同层冲突直接拒绝保存

3. 快捷执行时无当前会话
   - 提示“请先打开一个 SSH 会话”

4. 快捷执行时无命中脚本
   - 本地轻量提示“未找到脚本别名”

5. template 变量缺失
   - 参数卡片内联错误

6. 脚本被删除但终端候选缓存尚未更新
   - 执行前重新校验目标脚本是否仍存在

至少覆盖以下测试：

### 11.1 数据层

- alias 格式校验
- alias 分层唯一性校验
- 当前节点 alias 覆盖全局 alias

### 11.2 脚本库模型

- 列表搜索可命中 alias
- 高密度列表排序逻辑稳定
- 选中脚本切换不受大列表影响

### 11.3 终端快捷模型

- 只有整行开头 `x ` 才触发
- 300ms 延迟后才显示脚本候选
- 精确 alias 匹配能直接执行
- 未命中 alias 不会误执行
- 当前节点脚本优先于全局脚本

### 11.4 UI 交互

- 列表模式可浏览大量脚本
- 选择候选后 plain 脚本直接执行
- template 脚本会打开本地参数卡片
- 参数提交后最终命令正确发送到当前 session

## 12. 验收标准

本次设计完成后，应满足：

- 用户可在脚本库中为每个脚本维护 `alias`
- 用户可在终端中输入 `x alias` 快速运行脚本
- 输入 `x ` 300ms 后可见脚本推荐
- `plain` 与 `template` 脚本执行路径正确分流
- 当前节点脚本的 alias 解析优先于全局脚本
- 脚本库列表能在 20 到 30 条脚本场景下保持高效浏览
- `key` 与 `alias` 职责清晰分离

## 13. 取舍说明

本次有两个刻意取舍：

1. 未命中 `x alias` 时不透传原始命令到远端 shell
   - 目的是避免本地快捷命令失败后又向远端注入不可预期命令

2. template 参数卡片不接入 agent interaction 协议
   - 因为这是用户主动的本地脚本执行，不属于 agent runtime 的 HITL

这两个取舍保证本次迭代边界清晰，不把脚本库能力和 agent runtime 再次耦合。
