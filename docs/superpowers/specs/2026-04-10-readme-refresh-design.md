# OpsClaw README 重构设计

## 背景

随着 OpsClaw 功能持续扩展，现有 `README.md` 已经无法准确反映当前版本的能力边界与代码结构，主要问题包括：

- 部分内容已经过时，例如 Windows 打包产物版本号、脚本库入口描述等。
- 信息层级混杂，产品定位、快速启动、模块说明、FAQ、后续建议堆叠在同一层级，开发者难以快速建立全局认知。
- README 承担了过多细节说明，但 `docs/` 目录尚未形成清晰的导流关系。

本次迭代目标不是把 README 扩写成产品宣传页，而是把它重构为面向开发者/贡献者的入口文档：先帮助读者在 5 分钟内完成“理解项目 + 启动项目 + 找到代码入口”，再把深入内容导向 `docs/`。

## 目标

本次 README 重构的目标如下：

1. 让首次进入仓库的开发者快速理解 OpsClaw 当前是什么、主要解决什么问题。
2. 提供稳定且准确的本地启动方式，覆盖 Web 开发、桌面开发、Windows 打包三条主路径。
3. 按当前真实产品结构重写能力概览，补齐 AI、HITL、快捷脚本、节点状态 Dashboard、桌面运行时等新能力。
4. 提供清晰的架构概览和代码入口索引，降低贡献者定位模块的成本。
5. 明确 README 与 `docs/` 的职责边界，避免 README 持续膨胀。

## 非目标

以下内容不在本次 README 重构范围内：

- 不新增新的专题设计文档，只在 README 中为现有与后续 `docs/` 文档提供入口。
- 不把 README 写成面向外部访客的营销文案或完整产品白皮书。
- 不在 README 中保留过细的修复历史、零散 FAQ、未来建议清单。

## 读者与使用场景

### 主要读者

- 首次进入仓库的开发者
- 准备本地运行和调试的贡献者
- 需要快速定位模块入口的维护者

### 主要场景

- 想知道 OpsClaw 当前已经具备哪些核心能力
- 想在本地跑起 Web 或桌面开发环境
- 想了解系统由哪些核心模块组成
- 想继续深入阅读设计文档或实现计划

## README 结构设计

README 采用“快速上手 + 架构概览 + 文档导流”的双层结构，共分为以下一级章节。

### 1. OpsClaw 是什么

用途：

- 用一句话明确产品定位：AI-native SSH 运维工作台。
- 用一个短段落说明它为何对开发者重要：统一 SSH、AI、脚本、记忆、节点状态与桌面运行时，避免形成多个割裂工具。

约束：

- 不写空泛愿景。
- 不展开产品故事或路线图。

### 2. 快速开始

用途：

- 直接给出 `pnpm install`、`pnpm dev`、`pnpm desktop:dev`、`pnpm desktop:pack:win`。
- 明确当前版本号为 `0.2.0`。
- 说明当前主开发链路是 Web 开发与 Windows 桌面打包。

约束：

- 只保留高频命令。
- 不在这一节堆叠调试细节。

### 3. 当前能力概览

用途：

- 按用户工作流而非零散功能点分组，帮助开发者快速掌握系统全貌。

计划分组：

- 节点与终端
- AI Chat / Agent
- 结构化 HITL 交互
- 快捷脚本与 `x alias`
- 节点状态 Dashboard
- 设置与桌面运行时

约束：

- 每组只列核心能力，不写实现细节。
- 描述必须与当前代码行为一致，不能沿用旧 UI 结构名称。

### 4. 架构概览

用途：

- 建立顶层心智模型，说明前端工作台、服务端底座、Agent runtime、Electron 桌面壳和数据目录如何协作。
- 保留一个简化目录树。

约束：

- 只讲边界与职责，不展开内部实现流程。

### 5. 核心模块地图

用途：

- 给贡献者一个“从哪开始读代码”的路径索引。

计划覆盖的模块：

- Workbench 页面与布局编排
- AI 面板
- SSH 终端运行时
- 快捷脚本与设置面板
- 节点状态 Dashboard / 巡检服务
- Agent runtime
- HTTP API / WebSocket 网关
- Electron 主进程与预加载

约束：

- 以“模块职责 + 文件入口”为主，不做全量文件清单。

### 6. 开发与调试

用途：

- 收纳常用命令、数据目录、环境变量、桌面端调试提示。
- 用一句客观表述说明项目仍处于快速迭代中。

约束：

- 不保留大段 FAQ。
- 只保留高频、稳定、可执行的信息。

### 7. 文档索引

用途：

- 将深入内容导向 `docs/` 与 `docs/superpowers/`。
- 为未来专题文档留出扩展位。

约束：

- README 不再承载完整设计细节。

## 内容更新清单

本次改写至少需要修正或补齐以下内容：

- 将 Windows 打包产物版本号更新为 `0.2.0`。
- 移除对旧脚本库侧边面板的表述，改为设置中心中的脚本管理。
- 补充 AI Chat 与 Agent 双模式说明。
- 补充结构化 HITL 卡片交互说明，包括参数确认与执行审批。
- 补充 `x alias` 快捷脚本与 `x dashboard` 的终端交互入口。
- 补充节点状态 Dashboard、默认巡检、快照保留的能力说明。
- 强调桌面端为当前重要运行形态，Windows 打包为主要交付路径。

## README 与 docs 的职责边界

README 只负责以下三件事：

1. 解释项目是什么。
2. 指导开发者快速启动。
3. 提供架构与代码入口索引。

以下内容应通过链接下沉到 `docs/`：

- 详细设计文档
- 长篇实现计划
- 专题方案与演进记录

该边界用于避免 README 随功能迭代再次失控增长。

## 实施方式

本次实施采用直接重写 `README.md` 的方式，而不是在现有结构上做增量修补。原因如下：

- 现有 README 的章节设计已经不适合当前项目阶段。
- 增量修补会保留大量过时结构，长期成本更高。
- 本次目标是重建开发者入口文档，而不是做术语级修正。

## 风险与控制

### 风险

- README 可能再次变长，失去“快速入口”价值。
- 某些能力表述如果直接沿用旧认知，可能与当前实现不一致。
- `docs/` 当前内容不多，过度导流会造成跳转后信息不足。

### 控制措施

- 严格控制每节篇幅，优先写“能帮助理解和行动”的内容。
- 以当前代码结构为准更新模块与能力名称。
- 文档索引先链接已有文档，并为后续专题文档留出自然扩展点，不制造空链接。

## 验收标准

当以下条件满足时，本次 README 重构视为完成：

- 新读者能够在 5 分钟内理解 OpsClaw 的核心定位与主要能力。
- README 中的启动命令、版本号、打包路径与当前仓库一致。
- 开发者能够根据 README 快速定位主要模块入口。
- README 不再包含明显过时的 UI/模块描述。
- README 与 `docs/` 的职责边界清晰。
