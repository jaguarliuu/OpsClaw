# OpsClaw SFTP 文件管理器设计

> 日期：2026-04-13
> 状态：已确认设计，待进入 implementation planning

## 1. 背景

OpsClaw 当前已经具备：

- 节点管理与 SSH 连接能力
- 实时 terminal session 与 AI Agent 工作流
- 节点状态 dashboard、脚本库、HITL 审批与参数确认体系

但当前仍缺少一个面向“远端文件系统操作”的正式产品能力。用户希望在本周新增 SFTP 支持，并明确提出以下要求：

1. SFTP 是节点级正式能力，而不是临时附着在 shell 上的命令技巧
2. SFTP 与 terminal session 必须物理隔离，不能互相污染状态
3. 产品最终目标是完整的文件管理能力，而不是一次性的上传下载按钮
4. 高风险文件操作要兼顾效率与安全，遵循现有 HITL 审批体系
5. 文件传输链路需要从一开始就考虑稳定性、安全性和后续性能扩展

同时，当前 workbench 已暴露一个相关风险：多开 terminal session 后，界面会卡顿。根因不是单纯 Electron，而是当前 session 架构会为每个 tab 常驻完整 xterm runtime 和 websocket 连接。SFTP 设计必须避免沿用这种“多实例常驻”的模式，否则功能上线后会进一步放大性能问题。

## 2. 本轮范围

本次设计面向一个“架构按完整版设计、实现按阶段渐进落地”的 SFTP 能力。

### 2.1 最终目标形态

- 节点级 SFTP 文件管理器
- 单栏远端文件列表
- 顶部工具栏支持上传、下载、刷新、新建目录、重命名、删除、权限编辑、批量操作
- 右侧抽屉承载预览、元数据、权限编辑、传输详情
- 独立传输任务队列
- 支持暂停、恢复、重试、断点续传
- 复用节点 SSH 凭据，但保留 SFTP 单独覆盖认证能力
- 支持从当前 session 工作区快捷进入，但底层仍为独立连接

### 2.2 第一阶段目标

第一阶段优先交付稳定、可产品化的最小闭环：

- 节点级 SFTP 入口与页面框架
- 远端目录浏览
- 上传文件
- 下载文件
- 新建目录
- 重命名
- 删除
- 批量选择与批量删除
- 基础文件元数据展示
- 基础预览能力（文本优先，其他类型可降级）
- 独立传输队列
- 分块传输、失败重试、断点续传、临时文件提交
- 覆盖上传、批量删除、权限变更接入原生审批卡片

### 2.3 第一阶段非目标

本轮明确不做：

- 本地双栏文件树
- 常驻本地文件系统浏览器
- 多连接并行分段传输
- 目录拖拽上传
- 远端文件在线编辑
- 文件内容 diff
- 文件版本历史
- 全局跨节点传输中心
- 与 terminal 共用同一 websocket / runtime
- 让 Agent 直接操作交互式 SFTP shell

## 3. 设计原则

### 3.1 SFTP 是独立能力，不是 terminal 附属功能

SFTP 不能建立在现有 `session.run_command` 或 shell 注入协议之上。原因：

- shell 会话与文件传输协议语义完全不同
- terminal session 容易进入交互式子程序，状态不可控
- 文件操作需要结构化结果、任务状态、进度与审批，不适合走 shell 文本解析

因此本次设计将 SFTP 定义为节点级独立 runtime。

### 3.2 产品先按完整版设计，交付分阶段推进

尽管第一阶段不实现全部高级能力，但架构边界要直接面向最终目标：

- 右侧抽屉
- 任务队列
- 断点续传
- 权限编辑
- 性能治理

避免后续新增高级能力时推翻第一阶段实现。

### 3.3 安全与效率并重

采用用户已确认的原则：

- 单文件低风险操作可直接执行
- 批量删除、覆盖上传、权限修改等高风险操作必须先审批
- 关键参数不可由 Agent 自行编造
- SFTP 页面中的用户操作也应共享同一套原生审批卡片与 continuation 机制

### 3.4 稳定性优先于极限吞吐

第一阶段优先建立：

- 可恢复任务模型
- 分块传输
- checkpoint
- 临时文件提交
- 自动重试
- 暂停 / 恢复

并发分段与更激进的吞吐优化留到后续阶段。

### 3.5 不能放大现有 workbench 性能问题

SFTP 不能像当前 terminal 一样“每开一个节点或 tab 都常驻一个完整 runtime”。正确方向是：

- 只在用户真正打开 SFTP 视图时建立连接
- 同一时刻优先只维护当前活动节点的 SFTP 视图实例
- 切换节点时复用页面壳体，只切换数据源和连接上下文
- 不为每个 session 常驻一套文件管理组件

## 4. 产品交互设计

## 4.1 入口设计

SFTP 采用“双入口，但主入口为节点级”的产品策略：

1. 节点工作台一级视图
   - 与 `Terminal / AI / Dashboard` 平级
   - 是默认、正式入口

2. session 工作区快捷入口
   - 用户在当前节点 session 内，可快速切换到该节点的 SFTP
   - 仅作为导航捷径
   - 不代表 SFTP 与 session 共享运行时

### 4.1.1 与 session 的关系

- SFTP 与 terminal session 物理隔离
- 打开、关闭、重连 SFTP 不影响当前 shell
- shell 中断、前台交互程序变化，也不影响 SFTP

## 4.2 页面结构

SFTP 页面采用“单栏主列表 + 右侧抽屉”的结构，而不是双栏文件管理器。

原因：

- 用户明确反馈没有必要展示本地栏
- 本地选择文件可通过系统原生文件选择器完成
- 单栏结构更符合现有 workbench 视觉和交互习惯
- 可以显著减少本地文件系统状态与性能负担

页面组成如下：

1. 顶部工具栏
   - 当前路径面包屑
   - 刷新
   - 上传
   - 下载
   - 新建目录
   - 重命名
   - 删除
   - 权限编辑
   - 批量操作入口

2. 主文件列表区
   - 文件名
   - 类型图标
   - 大小
   - 修改时间
   - 所有者 / 组（可按阶段控制）
   - 权限摘要
   - 多选框

3. 右侧抽屉
   - 文件预览
   - 元数据
   - 权限编辑
   - 当前任务详情
   - 错误提示 / 重试入口

4. 传输队列区域
   - 可以作为右侧抽屉中的一个 tab，或固定次级区域
   - 展示进行中、暂停、失败、完成的任务

## 4.3 本地文件交互

本地文件不通过左侧文件树呈现，而通过系统原生能力接入：

- 上传：打开系统文件选择器
- 下载：打开系统保存位置选择器，或落到默认下载目录

这样可以显著降低前端复杂度，也更适合 Electron 桌面端体验。

## 4.4 典型用户流程

### 4.4.1 浏览远端目录

1. 打开节点 SFTP 视图
2. 系统建立或恢复该节点的 SFTP runtime
3. 默认加载初始目录
4. 用户点击目录项进行进入、返回和面包屑跳转

### 4.4.2 上传文件

1. 用户点击“上传”
2. 调起系统文件选择器
3. 用户选定本地文件
4. 如果目标文件不存在，直接创建传输任务
5. 如果目标文件已存在，先弹审批卡片
6. 审批通过后开始上传
7. 上传完成后刷新目录并选中新文件

### 4.4.3 批量删除

1. 用户多选文件
2. 点击“删除”
3. 系统生成批量删除审批卡片
4. 用户确认后执行
5. 完成后刷新目录

## 5. 架构设计

## 5.1 总体架构

SFTP 架构拆为以下核心模块：

### 5.1.1 `SftpConnectionManager`

职责：

- 管理按 `nodeId` 维度的独立 SFTP 连接
- 建立、复用、关闭、重连连接
- 默认从节点凭据读取 SSH 认证信息
- 支持 SFTP 独立覆盖认证配置
- 维护 host key 校验结果和连接状态

### 5.1.2 `SftpDirectoryService`

职责：

- 远端目录列表
- 文件 / 目录 `stat`
- 新建目录
- 删除
- 重命名
- 权限读取和基础权限修改
- 路径归一化与错误映射

### 5.1.3 `SftpTransferManager`

职责：

- 管理上传 / 下载任务队列
- 分块传输
- 进度更新
- checkpoint 写入
- 暂停 / 恢复 / 取消 / 重试
- 临时文件管理
- 传输完成校验

### 5.1.4 `SftpPolicyGate`

职责：

- 将高风险文件操作转换为统一的原生审批卡片
- 与现有 HITL / approval continuation 体系集成

### 5.1.5 `SftpSessionFacade`

职责：

- 向前端暴露统一、结构化的 API / event 协议
- 前端无需理解 ssh2/sftp 底层细节

## 5.2 与现有 terminal 架构的边界

终端与 SFTP 的边界必须清晰：

- terminal 负责 shell
- SFTP 负责远端文件系统
- 二者只共享：
  - 节点身份
  - 凭据来源
  - 审批体系
  - 设置与工作台导航
- 二者不共享：
  - websocket 运行时
  - 输入输出协议
  - 连接生命周期
  - 前端组件实例

## 6. 认证与连接策略

## 6.1 默认认证来源

用户已确认采用以下策略：

- 默认复用节点里已保存的 SSH 凭据
- 允许在 SFTP 面板里单独覆盖认证配置

这意味着：

- 普通场景下，用户打开 SFTP 即可连通
- 特殊场景下，SFTP 可以使用不同于 terminal 的账号或密钥

## 6.2 host key 校验

为避免“能连上就算成功”的不安全设计，SFTP 必须支持 host key 校验。

至少要满足：

- 首次连接时记录目标主机 host key 指纹
- 后续连接校验 host key 是否变化
- host key 异常变更时，不直接放行
- 需要用户显式确认或修复配置

## 6.3 连接生命周期

SFTP 连接生命周期应按“视图活跃度”管理：

- 打开视图时建立连接
- 视图空闲一段时间后可关闭或挂起
- 切换到其他节点时，优先释放非活跃节点连接
- 不要求每个节点常驻一条 SFTP 长连接

## 7. 文件传输与断点续传设计

## 7.1 任务模型

每个传输任务都必须是结构化、可恢复的。

状态枚举：

- `queued`
- `running`
- `paused`
- `retrying`
- `awaiting_approval`
- `failed`
- `completed`
- `cancelled`

任务字段建议如下：

```ts
type SftpTransferTask = {
  taskId: string;
  nodeId: string;
  direction: 'upload' | 'download';
  localPath: string;
  remotePath: string;
  tempLocalPath?: string;
  tempRemotePath?: string;
  totalBytes: number | null;
  transferredBytes: number;
  chunkSize: number;
  lastConfirmedOffset: number;
  status:
    | 'queued'
    | 'running'
    | 'paused'
    | 'retrying'
    | 'awaiting_approval'
    | 'failed'
    | 'completed'
    | 'cancelled';
  retryCount: number;
  errorMessage: string | null;
  checksumStatus: 'pending' | 'matched' | 'mismatch' | 'skipped';
  createdAt: string;
  updatedAt: string;
};
```

## 7.2 上传策略

上传不能直接写入目标文件，而应采用：

1. 创建远端临时文件
2. 以固定块大小分块写入
3. 每个块写入成功后更新 checkpoint
4. 全部传完后进行大小校验
5. 必要时执行校验和校验
6. 通过后原子 `rename` 到目标路径

这样可以避免：

- 网络中断留下半成品目标文件
- 权限冲突导致的部分覆盖
- 失败后无法恢复

## 7.3 下载策略

下载也采用相同原则：

1. 本地先写临时文件
2. 分块读取远端内容
3. 更新本地 checkpoint
4. 完成后校验
5. 最终 rename 为用户目标文件

## 7.4 断点续传

SFTP 的断点续传不采用 HTTP multipart，而采用基于 `offset` 的块续传。

规则：

- 每个任务都记录 `lastConfirmedOffset`
- 网络中断后从最后确认块继续
- 应用重启后可通过持久化任务记录恢复
- 如远端临时文件长度与本地 checkpoint 不一致，以一致性校验结果为准

## 7.5 重试与恢复

建议策略：

- 可配置的最大自动重试次数
- 短时网络错误自动退避重试
- 多次失败后转为 `failed`
- 用户可手动点击“恢复”或“重试”

## 8. 性能设计

## 8.1 第一阶段性能目标

第一阶段不追求极限吞吐，但要保证：

- 中大文件传输可持续稳定进行
- 不因单个任务卡死整个 UI
- 不为每个节点常驻完整 runtime
- 文件列表滚动、切目录、切节点保持流畅

## 8.2 传输吞吐策略

第一阶段：

- 单连接传输
- 固定块大小
- 按块确认与进度更新

后续阶段：

- 单连接下允许多个 in-flight 块
- 自适应块大小与窗口
- 针对高 RTT 链路提升吞吐

更后续阶段：

- 超大文件的多连接并行区段传输

本轮不直接做多连接并发，以控制复杂度和兼容性风险。

## 8.3 UI 性能约束

SFTP 前端必须遵守以下约束：

- 不为每个节点常驻一个完整文件面板实例
- 不在后台维护多个不可见目录树组件
- 大目录列表要考虑虚拟化能力
- 预览区异步加载，不能阻塞主列表
- 传输进度更新要节流，避免高频 state 抖动

## 9. 安全与审批模型

## 9.1 风险分级

低风险动作：

- 浏览目录
- 展开元数据
- 下载到本地
- 单文件上传到不存在的目标路径

高风险动作：

- 覆盖上传
- 批量删除
- 批量移动 / 重命名
- 权限修改
- 对系统关键目录进行写入

## 9.2 审批规则

遵循已确认策略：

- 单文件低风险动作可直接执行
- 高风险动作统一通过原生审批卡片

审批卡片应包含：

- 操作类型
- 节点信息
- 目标路径
- 影响范围
- 是否覆盖 / 权限变更值等关键参数

## 9.3 参数纪律

无论是用户直接操作还是后续 Agent 参与文件操作，都必须遵守：

- 目标路径不能臆造
- 权限值不能臆造
- 覆盖策略不能默认替用户决定
- 批量范围必须可见、可确认

## 10. 前端状态模型

建议拆分为以下几类状态：

### 10.1 `browserState`

- 当前节点
- 当前路径
- 文件列表
- 排序 / 筛选
- 选中项
- 列表加载态

### 10.2 `detailsDrawerState`

- 当前预览对象
- 元数据
- 权限编辑表单
- 当前选中任务详情

### 10.3 `transferQueueState`

- 活跃任务
- 暂停任务
- 失败任务
- 完成任务
- 聚合进度

### 10.4 `connectionState`

- `disconnected`
- `connecting`
- `ready`
- `degraded`
- `error`

### 10.5 `policyGateState`

- 当前待审批动作
- 审批卡片内容
- continuation payload

## 11. 异常处理

必须覆盖的异常包括：

- 连接中断
- 权限不足
- 路径不存在
- 远端文件在传输中被修改
- 本地目标文件已存在
- 校验失败

处理原则：

- 不让整个页面因为单个任务失败而变空
- 错误必须绑定到具体任务或具体路径
- 尽量提供恢复、重试、另存、覆盖等下一步动作

## 12. 分阶段实施建议

### Phase 1

- 节点级入口与页面框架
- 独立 SFTP runtime
- 目录浏览
- 上传 / 下载
- 新建目录 / 重命名 / 删除
- 基础右侧抽屉
- 任务队列
- 分块传输
- checkpoint
- 失败重试
- 断点续传
- HITL 审批接入

### Phase 2

- 权限编辑增强
- 文本 / 图片预览增强
- 大目录列表优化
- 传输性能优化
- 更完整的冲突处理

### Phase 3

- 更高级吞吐优化
- 大文件并发传输
- 在线编辑 / diff / 高级文件运维能力

## 13. 结论

本次 SFTP 设计的核心不是“在现有 shell 上再加一个上传下载按钮”，而是新增一个节点级、独立运行时的远端文件管理能力。

其关键结论如下：

1. SFTP 与 terminal session 物理隔离
2. 产品入口双通道，但主入口是节点级一级视图
3. 页面采用“单栏远端文件列表 + 右侧抽屉”，不做本地双栏
4. 上传下载从第一阶段就按“任务系统 + 分块传输 + 断点续传 + 临时文件提交”设计
5. 高风险操作统一接入现有原生审批卡片
6. 性能设计必须避免复制当前 terminal 多 session 常驻 runtime 的问题

这套设计既能支持本周开始落地，也为后续权限、预览、吞吐优化和更复杂的远端文件运维能力保留了稳定边界。
